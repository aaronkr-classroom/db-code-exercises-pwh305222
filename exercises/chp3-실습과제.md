# chp3-실습과제.md

# I. 데이터 요구 분석 명세서

### 1. 여행상품 관리

- 상품번호, 출발지, 도착지, 프로그램 정보, 시작일시, 종료일시, 최소 출발 인원, 최대 인원, 여행 경비, 예약금 정보, 출발여부

### 2. 예약 관리

- 예약번호, 고객ID, 상품번호, 예약날짜, 예약금 납부여부, 잔금 납부여부, 상태(확정/취소)

### 3. 고객 관리

- 고객ID, 이름, 주민등록번호, 휴대폰 번호, 주소, 고객등급

### 4. 직원 관리

- 사번, 이름, 주민등록번호, 휴대폰 번호, 급여, 담당업무, 입사일

# II. 트랜잭션 요구 분석 명세서

### 1. 여행상품 관리

- 상품 등록 : 새로운 여행상품 정보를 입력한다.(C)
- 상품 조회 : 계절별/상태별(판매 중) 여행상품 목록을 확인한다.(R)
- 출발 확정 : 출발 2주 전 인원 파악 후 출발 여부를 업데이트한다.(U)
- 배차 배정 : 출발이 결정된 상품에 관광버스와 운전기사를 1명씩 배정한다.(U)

### 2. 예약 관리

- 예약 신청 : 고객이 특정 여행상품에 대해 예약을 진행한다.(C)
- 예약금 확인 : 예약 후 24시간 이내 입금 확인 및 상태 변경, 미입금 시 예약 취소.(U)
- 잔금 결제 : 출발 1주일 전까지 잔금 납부 확인. 미납 시 예약 취소 및 예약금 몰수.(U)
- 환불 처리 : 여행상품 출발 취소 시 1주일 이내 고객에게 예약금을 반환한다.(U/D)

### 3. 고객 관리

- 고객 정보 관리 : 고객의 연락처, 주소 등 정보를 유지보수한다.(R/U)
- 등급 자동 업데이트 : 여행상품 참여 횟수에 따라 등급을 조정한다.(U)

### 4. 직원 관리

- 직원별 담당 확인 : 각 직원은 자신이 담당하는 여행상품과 해당 상품의 예약 현황을 조회한다.(R)
- 배차 현황 관리 : 운전기사 배정 및 버스 가동 상태를 관리한다.(R/U)

# III. 요구 수집 내용에 없는 세부 사항

### 1. 데이터 측면의 세부 사항

[고객 및 직원 보안]
- 비밀번호 : 앱을 통한 예약 시스템이므로 고객과 직원의 로그인 비밀번호가 저장되어야 합니다.
- 이메일 : 아이디 찾기나 예약 확인서 발송을 위해 필요합니다.
[결제 및 환불 기록]
- 결제 수단 : 카드, 계좌이체, 무통장 입금 등 결제 방식 정보가 필요합니다.
- 환불 계좌 정보 : 여행상품 취소 시 예약금을 돌려주기 위한 고객의 은행명과 계좌번호 정보가 필요합니다.
- 결제 일시 : '24시간 이내 결제'여부를 판단하기 위한 정확한 입금 시각 데이터가 저장되어야 합니다.
[관광버스 및 배차]
- 보험 정보 : 버스의 보험 가입 여부 및 만기일 관리 정보가 필요합니다.
- 정비 상태 : 차량의 안전을 위한 마지막 정비일 혹은 정기 점검 상태 데이터가 필요합니다.

### 2. 제약 조건 및 비즈니스 로직

[상태 값의 정의]
- 예약 상태 : 단순히 '예약'이 아니라 [예약대기 / 결제완료 / 잔금대기 / 확정 / 취소 / 환불완료]등 세분화된 상태 코드 정의가 필요합니다.
- 상품 상태 : [모집중 / 모집마감 / 출발확정/ 여행종료 / 취소] 등의 상태값이 필요합니다.
[예약 인원 제한]
- 잔여 좌석 계산 : (최대 인원 - 현재 예약 확정 인원)을 계산하여 남은 좌석이 0일 경우 예약을 차단하는 로직이 데이터베이스 레벨이나 애플리케이션 레벨에서 고려되어야 합니다.

### 3. 트랜잭션 세부 사항

[자동 취소 시스템]
- 배치 작업 : 예약 후 24시간이 지난 미결제 데이터를 찾아 자동으로 '취소' 상태로 변경하는 트랜잭션이 정의되어야 합니다.
[등급 산정 로직]
- 누적 횟수 계산 : 고객 등급(일반/우수/최우수)을 결정하기 위해 '참여 횟수'를 실시간으로 계산할 것인지, 아니면 특정 주기마다 업데이트할 것인지에 대한 정리가 필요합니다.

### 4. 확장성 고려

- 리뷰 및 별점 : 여행 상품 종료 후 고객이 남기는 후기와 평점 데이터가 필요할 수 있습니다.
- 쿠폰 및 이벤트 : 마케팅을 위한 할인 쿠폰 번호나 프로모션 적용 여부 속성이 필요할 수 있습니다.
- 다중 담당자 : 현재는 '상품 1개당 직원 1명'이지만, 대규모 상품의 경우 여러 직원이 협업할 가능성에 대비한 설계가 필요할 수 있습니다.
